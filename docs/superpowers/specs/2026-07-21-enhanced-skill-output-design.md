# Enhanced Skill Sample Output + Aligned Instructions

**Date:** 2026-07-21
**Status:** Approved (design), reference-in-progress

## Problem

Skill "Sample output" now renders in the Slack-style `DeliveryPreview` card (shared with
workflows). Two gaps remain:

1. **Visually plain.** No emojis; section headers are just `**bold**` lines. Wanted: richer,
   on-brand output using emojis, styled section headers, and (where needed) raw HTML.
2. **Instructions don't produce the sample.** Each skill's 5 platform prompts describe a
   *different* report format than its `sample_output`. Example: the External Company News
   Agent prompts describe a "Market Intelligence Report" (`### Executive Summary`,
   `### Company Overview`…) but the sample is a "Company News Brief"
   (`EARNINGS & FINANCIALS`, `LEADERSHIP CHANGES`, `M&A ACTIVITY`…). An agent following the
   prompt would never emit the shown output.

## Scope

All 30 skills in `web/public/skills.json` (and legacy `skills/skills.json` if separate). Both
the React app (`web/`) and legacy `index.html` renderers stay in sync.

## A. Shared "brief" markup

One format parsed by both React `SlackBody` (in `web/src/components/DeliveryPreview.jsx`) and
legacy `mdToHtml` (`index.html`). Existing tokens are unchanged; new tokens added.

| Token | Renders as |
|---|---|
| `## <emoji> SECTION LABEL` | `.so-section` — emoji + uppercase, tracked, muted, subtle top divider |
| `> Relevance: …` (blockquote line) | `.so-callout` — left accent bar, italic, muted |
| `===money===` | `.slack-money` green pill (existing) |
| `@mention` | `.slack-mention` blue chip (existing) |
| `**bold**` / `*em*` | `<strong>` / `<em>` (existing) |
| `- item` | `<ul><li>` (existing) |
| `---` | `<hr>` (existing) |
| Raw HTML | Passed through verbatim (trusted, authored content) |

### Renderer change

React `SlackBody` switches from building React nodes to building an **HTML string** using the
same logic as legacy `mdToHtml`, then rendering with `dangerouslySetInnerHTML`. This:
- unifies both codebases on one parser,
- enables the raw-HTML escape hatch for free,
- keeps content trusted (authored by us in `skills.json`, never user input).

Both parsers gain: `## ` → section header, `> ` → callout, and raw-HTML passthrough.

## B. CSS vocabulary

Added to `web/src/index.css` and legacy `<style>`, on-brand (Horizon Blue `#447C93` accent,
graphite text, Chivo Mono for uppercase labels):

- `.so-section` — section header row (emoji + uppercase label, `letter-spacing`, top border,
  margin-top). First section has no top border/margin.
- `.so-callout` — left `border-left` accent in Horizon Blue, italic, muted text, small pad.
- Reuse existing `.slack-money`, `.slack-mention`.

## C. Sample content pattern

Each `sample_output.content` follows:

```
**<Brief Title>: {Account}**

## <emoji> SECTION ONE
- **Item (date):** fact with ===$metrics=== and @mentions.
- **Item:** fact.
> Relevance: why this matters for the seller.

## <emoji> SECTION TWO
…

---
*Generated by Backstory <Agent> — <n> sources analyzed …*
```

Emoji guide (consistent across catalog): 📊 financials/metrics · 👥 people/leadership ·
🚀 product/strategy · 🤝 partnerships/M&A · 🔔 signals/alerts · ⚡ actions · 🎯 priorities ·
⚠️ risk · 📅 timing · 💰 deal/pipeline. Pick per section meaning.

## D. Instructions alignment

For each skill, rewrite **only the output-format section** of all 5 platform prompts
(`claude-project`, `claude-code`, `chatgpt-gpt`, `copilot`, `gemini`) so the described output
structure matches its `sample_output`: same section names + emojis, brief style, `> Relevance:`
lines, and the footer. Leave process/tools/rules sections intact. The agent's live output is
plain Slack/markdown text — parity means *structure and content*, not our HTML classes.

## Execution

1. **Reference:** build renderer + CSS, then do the External Company News Agent (skill #02)
   sample + 5 prompts end-to-end. Get visual approval.
2. **Fan-out:** orchestrated multi-agent workflow — one agent per remaining skill rewrites its
   sample + 5 prompts to the approved pattern, output validated against the skills.json schema.

## Non-goals

- No change to the Prompt/Platforms tab's copy buttons (those prompts are meant to be copied).
- No new dependencies. No sanitizer library (content is trusted).
- No refactor of unrelated skill fields.
