About
The Backstory Automation Library

Reusable, implementation-ready workflow patterns for revenue teams — not one-off builds.

This library contains ready-to-import workflow automations that leverage the Backstory MCP (Model Context Protocol) integration alongside AI models, CRM data, and communication tools to deliver actionable sales intelligence.

Each workflow is packaged as a reusable pattern rather than a one-off customer build. The library combines validated implementations, recipes for the most common orchestration tools, and generic adaptation guidance so the same workflow can move across different CRM, delivery, and source-system environments.

Packaging Strategy:
Validated: n8n + orchestrator instructions Recipes: Make / Power Automate / Zapier Adaptable: other orchestration stacks

Cross-Tool Template Model:
1. Start from the workflow pattern and pick the closest validated implementation
2. Use the recipe card if the customer is on Make, Power Automate, Zapier, or a similar orchestrator
3. Swap equivalent connectors for CRM, delivery, notes, and data sources while preserving the workflow shape
4. Productize the logic that repeats across customers: prompts, scoring, routing, and thresholds
5. Keep vendor-specific steps thin so the workflow survives stack changes

Common System Families:
• CRM / system of record: Salesforce, Dynamics 365, HubSpot, or a customer-specific CRM/data warehouse
• Delivery: Slack, Microsoft Teams, email, or internal queues
• Meeting / note sources: Google Calendar, Microsoft 365, Gong, Zoom, Otter, Fireflies, Fathom
• Orchestration: n8n, Make, Power Automate, Zapier, Workato, or custom code

Maintained by: AI Innovation Team
Slack: #automation-workflows

API Reference

API reference

Every Backstory API endpoint, grouped by resource. All endpoints use host https://api.people.ai and bearer-token authentication — send Authorization: Bearer <access_token> on every request.

▶

Run your first call (60 seconds). Mint a token from your API key and secret, then read an Engagement Level snapshot end to end.

Read path vs. write path

Credentials are production — there is no sandbox.

  • GETs are safe to run against your org.
  • Write paths mutate or schedule work: activity ingestion (PUT /v0/public/activities/email) and bulk export (POST /pull/v1/export/...). Run with intent.
⚠

Every POST/PUT here is a live write. Stick to GETs for exploration.

Resources

GroupWhat it covers
AuthenticationMint a bearer token from your API key and secret.
Accounts, Contacts, Leads, OpportunitiesRead CRM records by id, CRM id, or email.
ActivitiesRead emails, calls, and meetings, plus their participants.
Teams, Team membersRead your sales hierarchy and users.
EngagementRead the Engagement Level over time.
Bulk exportAsynchronous export jobs for raw activity data and enriched contacts.

Insights (Beta)Export metric tables with the Query API.

Conventions

  • Auth: Authorization: Bearer <access_token> on every request.
  • Pagination: list endpoints take offset and limit; responses include meta.
  • Envelope: payload in data; lists also include meta.
  • Errors: one envelope across all statuses. Engagement endpoints reject single-day or inverted ranges with 422; the Query API silently drops unknown slugs with 200. See Errors.

See Rate limits and conventions.

API Reference

Rate limits and conventions

The REST data API shares a consistent set of conventions across every resource.

Rate limits

LimitValue
Maximum requests per second5
Maximum concurrent requests10
Maximum page size100,000
Maximum date range for activity data2 years

The REST API does not return X-RateLimit-* headers. Stay within the limits above, and retry with backoff if you receive a 429.

ⓘ

The Beta endpoints have their own limits. The Query API allows roughly 100 requests per hour on a rolling window; see Run an insights query.

Pagination

List endpoints accept offset and limit query parameters and return a meta object alongside the data array. The default limit is 1000; the maximum is 100,000.

curl "https://api.people.ai/v0/public/accounts?offset=0&limit=500" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
{
  "data": [ { "id": 140089, "name": "Northwind Logistics" } ],
  "meta": { "limit": 500, "offset": 0, "total": 2843, "message": null }
}

meta.message is an optional advisory (for example, a deprecation notice); it is null when there is nothing to report. To page through a full result set, increase offset by limit until offset reaches meta.total.

Response envelope

Every REST response wraps its payload in a data field:

  • List endpoints return data as an array, with a meta pagination object.
  • Single-record endpoints return data as an object, with meta set to null.

Dates and timestamps

  • Query parameters that take a date (from, to, ingested-from, ingested-to) use the YYYY-MM-DD format. Use last-n-days for a relative window, and time-zone to set the interpretation of the range.
  • Timestamp fields in responses (for example, activity_timestamp, created_at, ingested_at) are Unix epoch milliseconds.
  • Request bodies for activity ingestion, and Beta filter dates, use Unix epoch seconds.

Errors

Errors use a consistent envelope:

{
  "code": 401,
  "id": "0f9a3c7e1b5d4826ad03e9f1c7b62d54",
  "message": "Invalid or expired token",
  "path": "/v0/public/accounts",
  "timestamp": "03-06-2026 03:25:25"
}

The id is a unique per-request trace identifier (a hex hash) that changes on every response — log it when reporting a problem, but do not branch on its value. timestamp is the server time the error occurred, formatted DD-MM-YYYY HH:MM:SS.

StatusMeaning
400Malformed request — for example, an out-of-range limit or a malformed job id.
401Missing or invalid bearer token. Mint a new token.
404No record matches the requested id.
422Invalid filter range — a single-day or inverted date range on the Engagement endpoints. Pass from strictly before to.
429Rate limit exceeded. Back off and retry; no Retry-After header is returned.
500An unexpected server error. Retry with backoff.

For 429 and 500, retry with exponential backoff. Because no Retry-After header is returned, choose your own base delay — for example, start at one second and double it up to a cap.

For the full set — including the Query API failures that return HTTP 200 while silently dropping a column — see the Errors reference.

Versioning and Beta

Backstory API endpoints are versioned in their path. Different capabilities sit at different versions.

Path prefixCapabilityStatus
/auth/v1/tokensAccess tokens for the REST and bulk APIsGA
/v0/publicREST data API and activity ingestionGA

/v3/auth/tokensAccess tokens for the Beta endpointsBeta
/v3/beta/insights/exportInsights export and Query APIBeta
/v3/beta/forecasting/queryForecasting Query APIBeta

GA endpoints

General availability (GA) endpoints are stable. Backstory adds fields and capabilities in a backward-compatible way. Treat unknown fields in responses as additive and ignore the ones you do not use.

Beta endpoints

⚠

Beta endpoints under /v3/beta can change. Request and response shapes, available metrics, and limits may evolve before the endpoints reach GA. Pin to documented behavior and revalidate before depending on a Beta endpoint in production.

Beta endpoints also authenticate differently from GA. They use the POST /v3/auth/tokens token endpoint, though their tokens share the same 7200-second default lifespan as GA.

API Reference

Data model

The REST API exposes the core go-to-market objects Backstory maintains. This page explains how they relate and how to reference a record.

Objects

ObjectDescription
AccountA company in your CRM.
OpportunityA deal, linked to an account.
ContactA known person at an account.
LeadAn unconverted prospect. When converted, it links to an account, contact, and opportunity.
ActivityAn email, call, or meeting, with participants. Activities match to accounts and opportunities.
TeamA node in your sales hierarchy.
Team memberA user, who belongs to a team.
Engagement LevelA 0–100 score for an account, contact, lead, or opportunity, based on the volume, recency, and type of matched activities.

Referencing records

Every record carries two identifiers:

  • id: Backstory's internal numeric id.
  • crm_id: the id of the record in your source CRM (for example, Salesforce).

Most resources offer a lookup by each. For example, an account is available at /v0/public/accounts/{id} and at /v0/public/accounts/crm-id/{crmId}. Contacts, leads, and team members also support lookup by email.

Relationships

  • An activity carries account_id / account_crm_id and, when matched, opportunity_id / opportunity_crm_id. Backstory matches activities to CRM records using signals such as participant domains and opportunity contact roles. See Matching in the help center.
  • An activity participant describes a person on an activity, with account_id, lead_id, opportunity_id, and contact attributes such as job_title and seniority.
  • A team member references its team_id, team_crm_id, and parent_team_id. Use the team-and-descendants endpoints to read a team's full subtree.

Engagement Level

Backstory recomputes Engagement Level on a 24-hour cycle. The endpoints return a score history: a list of point-in-time snapshots, each { id, crm_id, score, date, timestamp }. date is the score date (YYYY-MM-DD); timestamp is that date at midnight UTC, in Unix epoch milliseconds. Every record is a snapshot, not a live value.

Snapshot frequency depends on the object:

  • Opportunities return a daily snapshot, from the opportunity's creation through its close.
  • Accounts, contacts, and leads return a weekly snapshot, covering roughly the last three months.

Query the organization-wide endpoint for each object type, optionally filtered to a single record with the matching *-id parameter. Set from and to to a range spanning at least one day.

Errors

Read this before you ship a job: it lists every status the API returns, the error envelope you parse, and the three Query API traps that return HTTP 200 while silently dropping your data.

Error envelope

Every error shares one envelope:

{
  "code": 401,
  "id": "0f9a3c7e1b5d4826ad03e9f1c7b62d54",
  "message": "Invalid or expired token",
  "path": "/v0/public/accounts",
  "timestamp": "04-06-2026 03:25:25"
}

id is a unique per-request trace identifier (a hex hash) that changes on every response — log it when reporting a problem, but never branch on its value. timestamp is the server time the error occurred, formatted DD-MM-YYYY HH:MM:SS.

Status codes

StatusMeaningWhat to do
400Malformed request — an out-of-range limit, a malformed job id, an invalid body.Fix the request; do not retry unchanged.
401Missing or invalid bearer token.Mint a new token.
404No record matches the requested id.Check the id; expected for unknown records.
422Invalid filter range — a single-day or inverted date range on the Engagement endpoints.Pass from strictly before to. See below.
429Rate limit exceeded. No Retry-After header.Back off and retry — start at 1s and double up to a cap.
500Unexpected server error.Retry with exponential backoff.

The from < to rule (422)

The Engagement endpoints require from to be strictly before to. A single-day range (from == to) or an inverted range (from > to) returns 422, not an empty result:

Single-day range — returns 422
curl "https://api.people.ai/v0/public/accounts/insights/engagement?from=2026-06-01&to=2026-06-01" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
422 — Invalid date filters range
{
  "code": 422,
  "message": "Invalid date filters range",
  "path": "/v0/public/accounts/insights/engagement"
}

A valid range (from=2026-05-01&to=2026-06-01) returns 200.

Failures that return 200 (read this)

The most dangerous failures on the Query API and REST endpoints are not 4xx — they return HTTP 200 and silently drop data. A typo becomes an invisible data-quality bug, not an error.

⚠

Unknown query params, unknown export columns, and invalid metric slugs are accepted with HTTP 200 and silently ignored — they do not error. Always validate every slug and param name against the live API before you depend on a job's output.

Invalid slug → empty column, no error

On POST /v3/beta/insights/export, a bogus ootb_ slug returns 200 with the column omitted. A valid anchor column plus a bogus one returns 200 carrying only the anchor — the bad slug vanishes silently:

Anchor + bogus slug
curl -X POST https://api.people.ai/v3/beta/insights/export \
  -H "Authorization: Bearer YOUR_BETA_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "object": "opportunity",
    "columns": [
      { "slug": "ootb_opportunity_engagement_level" },
      { "slug": "ootb_opportunity_THIS_DOES_NOT_EXIST_zzz" }
    ]
  }'
Response — 200, only the valid column survives
Opportunity Engagement Level
71
58
...

The ootb_opportunity_THIS_DOES_NOT_EXIST_zzz column is gone, with no warning. Diff your returned columns against the columns you requested on every run.

trend_by is unsupported on export → 400

trend_by is recognized but rejected on the export endpoint. It returns the only hard error in this section:

400 — trend_by on /v3/beta/insights/export
{
  "message": "Trend by unsupported in exporting"
}

The export returns the current snapshot value per row, not a time series. For Engagement Level history, use the dedicated endpoints GET /v0/public/{accounts|contacts|leads|opportunities}/insights/engagement.

      Use this guide when a workflow needs to move from a single-customer implementation
      to a repeatable offering that works across multiple customer environments.
    


    
The Three-Layer Model

    

      
        

          Layer
          What to Ship
          Why It Exists
        

      
      
        

          Validated implementations
          n8n JSON, Claude workflow instructions, OpenAI workflow instructions
          These are the fastest starting points when the customer stack matches a known-good implementation.
        

        

          Deep recipes for common orchestrators
          Make, Power Automate, Zapier, and similar step-by-step rebuild guides
          These cover the most common orchestration environments without trying to support every tool equally.
        

        

          Generic adaptation path
          Connector substitution guidance for customer-specific stacks
          This is how the workflow survives differences in CRM, delivery, note takers, and orchestration tooling.

  • The workflow pattern: trigger → gather → enrich → analyze → route → deliver
    • 
      
  • Prompts, scoring logic, thresholds, and escalation rules
    • 
      
  • Canonical payload shapes between steps
    • 
      
  • Configuration knobs like lookback windows, channel routing, account filters, and owner mappings
    •

  • Vendor-specific connector steps
    • 
      
  • Authentication wrappers
    • 
      
  • Field-name mappings between Salesforce, Dynamics, HubSpot, or custom schemas
    • 
      
  • Delivery-node implementations for Slack, Teams, email, or queueing tools

Common Substitution Matrix

    

      
        

          System Family
          Common Choices
          Adaptation Rule
        

      
      
        

          CRM / system of record
          Salesforce, Dynamics 365, HubSpot, custom CRM, data warehouse
          Swap fetch and write-back steps, but keep the canonical account, contact, opportunity, and activity payloads the same.
        

        

          Delivery
          Slack, Microsoft Teams, email, ticket queues
          Replace only the final routing and formatting layer; do not fork the business logic.

          Normalize meetings, transcripts, and attendees into a shared enrichment payload before the AI step.

          Preserve workflow order and state handling even if the tool surface changes.
        

      
    


    
Prioritization Rule

    

      Do not try to write recipes for every tool. Prioritize the ecosystems that are both
      common in the field and validated through real implementations. In practice, that means:
    

    
    
      
  1. Ship deep support for the most common orchestrators first.
    2. 
      
  2. Reuse workflows already proven in customer implementations as the template baseline.
    3. 
      
  3. Use the generic adaptation layer for less common customer-specific stacks.
    4. 
    


    

      Practical rule: Productize the pattern, not the vendor. The more the prompts,
      decision logic, and payload contracts can survive a connector swap, the more reusable the
      workflow becomes across customers.

Backstory MCP Setup Guide

    

      The Backstory Model Context Protocol (MCP) server is the core data source
      for every workflow in this library. It provides AI agents with direct access to account
      activity, engagement data, opportunity status, and contact intelligence.
    


    

      What is MCP? Model Context Protocol is an open standard that lets AI agents
      call external tools and data sources. Backstory's MCP server exposes sales intelligence
      as tools that agents can invoke during reasoning — no custom API integration code needed.
    


    

      Prerequisites: An n8n instance (cloud or self-hosted v1.19.0+) and an active Backstory account with MCP API access enabled. The Backstory n8n instance is https://n8n-pg.peoplesync.ai.
      For the official Backstory guide, see Connect n8n to Backstory.
    


    
    
What You Can Do

    
    
      
  • Automate sales intelligence workflows like daily account health checks, weekly opportunity reviews, or risk alerts
    • 
      
  • Build AI-powered chatbots for Slack or Teams that answer questions with real CRM data
    • 
      
  • Create coaching automation that analyzes opportunities and generates weekly insights
    • 
      
  • Generate automated reports combining Backstory insights with data from other systems
    • 
      
  • Trigger actions based on sales data like sending alerts when accounts show low engagement
    •

Troubleshooting

    

      
        

          Error
          Cause
          Fix
        

      
      
        

          535 Authentication failed
          Wrong password or App Password needed
          Use App Password (Gmail) or enable SMTP AUTH (M365)
        

        

          Connection refused (port 25)
          ISP or cloud provider blocks port 25
          Use port 587 (STARTTLS) or 465 (SSL) instead
        

        

          Emails going to spam
          Missing SPF/DKIM/DMARC records
          Set up domain authentication with your email provider
        

        

          Daily sending limit reached
          Exceeded provider quota
          Upgrade plan or switch to SendGrid/SES for volume
        

        

          HTML not rendering
          Sent as plain text instead of HTML
          Set content type to text/html or use multipart/alternative
        

      
    


    

      Security Tip: Never use your primary account password for SMTP.
      Always use App Passwords, API keys, or OAuth tokens. Store credentials in environment
      variables or your platform's secret store.
